# MyHWManager

An open-source, lightweight hardware monitoring and remote management platform built with a Client-Server architecture. Monitor hardware telemetry in real-time, manage files remotely, and control devices through a web-based terminal and remote desktop — all from a single dashboard.

> **⚠️ Project Status: Active Development / Prototype**  
> This project is functional but still a work-in-progress. Some features from the original vision are implemented and working, while others remain planned. This README is honest about both.

## The Idea

The original vision was to build a comprehensive, lightweight, open-source tool for managing personal computers and small server clusters — something like a self-hosted mix of HWiNFO + RustDesk + Google Drive + a dash of Ansible. The full wishlist included:

- ✅ Full hardware inventory and live telemetry (CPU, RAM, disk, network, temperatures)
- ✅ Multi-device management through a central server with grouping
- ✅ Remote terminal access (SSH-like, browser-based)
- ✅ Remote desktop viewing and control
- ✅ Centralized file storage ("Central Hub") with per-device sandboxed file browsing
- ✅ Secure device pairing via one-time PINs
- ✅ Multi-language UI (English, Vietnamese, Russian, Chinese, Japanese)
- ❌ Driver detection, update, and installation
- ❌ Notification bots for Discord/Telegram
- ❌ Wake-on-LAN via ESP32/Arduino
- ❌ Built-in firewall management
- ❌ Low-latency remote desktop suitable for gaming
- ❌ Hardware change history tracking over time
- ❌ Cross-platform client builds (currently Linux-focused)

The project started as a personal tool and grew from there. It's honest code from a real development process — not a polished product, but a working prototype that does useful things.

## What's Been Built

These features are implemented and functional:

### 🖥️ Live Hardware Telemetry
The Tauri-based client collects real-time system data using the `sysinfo` crate (Rust) and streams it to the server over WebSockets. The admin dashboard shows live CPU usage per-thread, memory consumption, disk partitions, network throughput, and thermal sensor readings.

### 🔗 Secure Device Pairing
Devices connect to the server using a 6-digit one-time PIN generated by the admin. PINs expire after 10 minutes, are single-use, and the system enforces rate-limiting (5 failed attempts = 30-minute lockout per IP). After pairing, each device receives an HMAC-SHA256 signed agent token for persistent authentication.

### 📂 File Server (Central Hub)
A shared file storage space hosted on the server. Both admin and paired clients can upload, download, create folders, rename, move, and delete files through a Google Drive-inspired UI. Files are stored in a `storage/` directory on the server with metadata tracked in SQLite.

### 📂 Remote File Explorer (Client Filesystem)
The admin can browse files on a client's machine directly from the dashboard. **This is sandboxed** — the client user explicitly chooses which directories the admin can access. The admin cannot see or touch anything outside those allowed paths. All filesystem operations (list, create, rename, move, upload, download, delete) are validated against the allowlist on the client side in Rust before execution.

### 💻 Remote Terminal
A browser-based terminal that opens a real PTY (pseudo-terminal) session on the client machine. The admin types in an xterm.js terminal in the browser; keystrokes are relayed via WebSocket to the client, which runs them in a local shell. Output streams back in real-time. Sessions are cleaned up on disconnect.

### 🖵 Remote Desktop
A basic remote desktop implementation where the client captures screen frames and streams them to the admin's browser via a WebSocket relay. The admin can send mouse clicks and keyboard input back. This works, but it is **not** low-latency — it's adequate for administrative tasks, not for gaming or video playback.

### 🌐 Multi-Language Support
The entire UI (both server dashboard and client app) supports hot-swappable language switching via `react-i18next`. Currently supported: English, Tiếng Việt, Русский, 中文, 日本語.

### ⚙️ Admin Dashboard
A full React-based admin panel with login, device overview, device detail pages, activity logs, settings (log retention period), and device group management.

## What's Not Done Yet

Being honest about the gaps:

| Feature | Status | Notes |
|---------|--------|-------|
| **Driver management** | ❌ Not started | Was in the original vision but requires OS-specific backends and vendor API integrations. Significant scope. |
| **Discord/Telegram bots** | ❌ Not started | Planned for sending alerts when devices go offline or hardware issues are detected. |
| **Wake-on-LAN (ESP32)** | ❌ Not started | Requires separate embedded firmware and network broadcast support. |
| **Firewall management** | ❌ Not started | Would need root/admin privileges and OS-specific iptables/nftables/Windows Firewall integration. |
| **Low-latency remote desktop** | 🟡 Partial | Current implementation works but uses MJPEG-over-WebSocket, not WebRTC peer-to-peer. Noticeable latency, not suitable for gaming. |
| **Hardware change tracking** | ❌ Not started | The data model stores a snapshot of hardware at pairing time but doesn't track changes over time. |
| **Windows/macOS client** | 🟡 Partial | The Tauri app is cross-platform in theory, but GPU detection (`lspci`) and some terminal behavior are Linux-specific. Untested on other platforms. |
| **HTTPS/TLS** | ❌ Not implemented | All communication is over plain HTTP/WS. For production use, you'd need a reverse proxy (Nginx/Caddy) with TLS. |
| **Multi-admin support** | ❌ Single admin only | The system supports only one admin account. Multi-user admin with role-based access is not implemented. |
| **File sharing between clients** | 🟡 Schema exists | The `FileShare` database model exists but the sharing UI and API logic are not wired up. |

## Architecture

```
┌──────────────────────────────────────────┐
│              Admin Browser               │
│  (React + TailwindCSS + xterm.js)        │
│  Server Frontend — port 5173             │
└──────────┬───────────────────────────────┘
           │ HTTP REST + WebSocket
           ▼
┌──────────────────────────────────────────┐
│           Go Backend (Gin)               │
│  REST API + WebSocket Relay              │
│  SQLite DB + File Storage                │
│  Server Backend — port 8080              │
└──────────┬───────────────────────────────┘
           │ WebSocket (telemetry, terminal, RDP, filesystem relay)
           ▼
┌──────────────────────────────────────────┐
│         Tauri Client (Rust + React)      │
│  sysinfo telemetry · PTY terminal       │
│  Sandboxed filesystem · Input relay      │
│  Desktop app — port 1420 (dev)           │
└──────────────────────────────────────────┘
```

**Communication model:**
- **Telemetry:** Client → Server via persistent WebSocket (`/ws/ingest/:id`). JSON frames every ~1 second.
- **Terminal:** Bidirectional WebSocket relay (`/ws/terminal/signal/:id`). Admin ↔ Server ↔ Client PTY.
- **Remote Desktop:** Bidirectional WebSocket relay (`/ws/rtc/signal/:id`). Screen frames + input events.
- **File Server (Hub):** Standard REST API with multipart upload. Files stored on server disk.
- **Remote File Explorer:** Server acts as relay — sends filesystem commands to client via existing telemetry WebSocket, client responds with results.

## Getting Started

### Prerequisites

Make sure you have these installed:

- **Go** ≥ 1.21 — [install guide](https://go.dev/doc/install)
- **Node.js** ≥ 18 — [install guide](https://nodejs.org/)
- **Rust** (stable) — [install guide](https://rustup.rs/)
- **Tauri v2 prerequisites** (Linux) — system libraries for WebKitGTK:
  ```bash
  # Ubuntu/Debian
  sudo apt install libwebkit2gtk-4.1-dev build-essential curl wget file \
    libxdo-dev libssl-dev libayatana-appindicator3-dev librsvg2-dev

  # Fedora
  sudo dnf install webkit2gtk4.1-devel openssl-devel curl wget file \
    libxdo-devel libappindicator-gtk3-devel librsvg2-devel
  ```

### Running the Server

The server has two parts: a Go backend and a React frontend.

**Terminal 1 — Backend:**
```bash
cd server/backend
cp .env.example .env   # Or edit the existing .env (see Configuration below)
go run .
```
The backend starts on `http://localhost:8080`.

**Terminal 2 — Frontend:**
```bash
cd server/frontend
npm install            # First time only
npm run dev
```
The admin dashboard opens at `http://localhost:5173`.

**Default login:** `admin` / `admin` (change this in `.env`).

### Running the Client

```bash
cd client
npm install            # First time only
npm run tauri dev
```

This builds the Rust backend and launches the Tauri desktop app. The client UI appears as a native window.

To connect to a server:
1. On the server dashboard, go to Settings and click **Generate PIN**.
2. On the client, click **Link Server**, enter the server's IP:port (e.g., `192.168.1.100:8080`) and the 6-digit PIN.
3. The client pairs and begins streaming telemetry.

### Quick Start Script

There's a convenience script to start the entire server (backend + frontend) at once:

```bash
cd server
chmod +x run.sh
./run.sh
```

Press `Ctrl+C` to stop both services.

## Configuration Reference

### Server `.env` File

Located at `server/backend/.env`:

```env
# ==========================================
# MyHWManager Server Configuration
# ==========================================

# Admin credentials (used to create/reset the admin account on startup)
# The admin account is synced from these values EVERY time the server starts.
# Changing these and restarting will update the admin password.
ADMIN_USERNAME=admin
ADMIN_PASSWORD=admin

# JWT Secret for agent token generation.
# Agent tokens are HMAC-signed using this secret. If you change it,
# ALL previously paired devices will need to re-pair.
# Use a long random string in production.
JWT_SECRET=myhwmanager-secret-change-in-production

# Server Port — the Go backend listens on this port.
SERVER_PORT=8080
```

**Important notes:**
- `ADMIN_USERNAME` / `ADMIN_PASSWORD`: These are synced to the database on every startup. If you change them and restart, the admin credentials update immediately. There is no "forgot password" flow — just edit `.env` and restart.
- `JWT_SECRET`: This is used to generate agent tokens for paired devices. **Do not change this after devices are paired**, or they will lose authentication and need to re-pair. Admin session JWTs use a separate random secret that regenerates on every restart (meaning admin sessions expire on server restart — this is intentional).
- `SERVER_PORT`: Defaults to `8080` if not set.

### Server Frontend Environment

The frontend reads API URLs from Vite environment variables. For development, defaults are used. For production or non-standard setups, create a `.env` file in `server/frontend/`:

```env
VITE_API_URL=http://your-server-ip:8080
VITE_WS_URL=ws://your-server-ip:8080
```

### Client Configuration

The client has no `.env` file. Configuration happens through the UI:
- **Server address:** Set when clicking "Link Server" in the app header.
- **Allowed directories:** Managed in the "Files" tab — you add/remove which local folders the admin can access remotely.
- **Language / Theme:** Set in the header toolbar and persisted in localStorage.

Internally, the client stores:
- `client_id`: A randomly generated UUID, stored in localStorage. Identifies this device.
- `agent_token`: Received from the server after pairing, stored in localStorage.
- `allowed_dirs.json`: Stored in the Tauri app config directory (`~/.config/com.ayamiko.client/`). Lists absolute paths the admin can browse.

### CORS Configuration

The server backend has hardcoded allowed origins for CORS and WebSocket upgrades:
```
http://localhost:5173   (Vite dev server)
http://localhost:1420   (Tauri dev server)
http://localhost:8080   (Backend self)
tauri://localhost       (Tauri production)
```

**If you deploy to a different host or port**, you must update the `allowedOrigins` arrays in `server/backend/main.go` (two places: the `upgrader` variable and the `cors.New` configuration). This is a known limitation — it should be environment-configurable but currently isn't.

## Deployment Guide (Small Scale)

This setup is suitable for a home lab, small office, or personal use with a handful of machines.

### 1. Prepare a Server Machine

Any Linux machine on your local network will work (a Raspberry Pi, an old laptop, a VM, etc.).

```bash
# Clone the repo
git clone https://github.com/ayamiko/MyHWManager.git
cd MyHWManager

# Configure the server
cd server/backend
nano .env    # Set a strong admin password and JWT secret
```

### 2. Build and Run the Server

```bash
# Build backend
cd server/backend
go build -o myhwmanager_backend .

# Build frontend
cd ../frontend
npm install
npm run build

# Start backend (it can serve the frontend's dist/ too if configured,
# but for simplicity, run them separately or use the run.sh script)
cd ..
./run.sh
```

### 3. (Recommended) Put Behind a Reverse Proxy

For anything beyond localhost, use Nginx or Caddy to:
- Add HTTPS/TLS
- Proxy WebSocket connections properly
- Serve the frontend statically

Example Nginx config:
```nginx
server {
    listen 443 ssl;
    server_name hw.yourdomain.local;

    ssl_certificate     /path/to/cert.pem;
    ssl_certificate_key /path/to/key.pem;

    # Frontend
    location / {
        proxy_pass http://127.0.0.1:5173;
    }

    # Backend API + WebSocket
    location /api/ {
        proxy_pass http://127.0.0.1:8080;
    }

    location /ws/ {
        proxy_pass http://127.0.0.1:8080;
        proxy_http_version 1.1;
        proxy_set_header Upgrade $http_upgrade;
        proxy_set_header Connection "upgrade";
        proxy_read_timeout 86400;
    }
}
```

### 4. Install Client on Target Machines

Build the client on each machine, or distribute the pre-built binary:
```bash
cd client
npm install
npm run tauri build
# Install the .deb/.rpm/.AppImage from src-tauri/target/release/bundle/
```

Then pair each client using a PIN from the admin dashboard.

### 5. Keep It Running

Use `systemd` to run the server backend as a service:
```ini
# /etc/systemd/system/myhwmanager.service
[Unit]
Description=MyHWManager Server
After=network.target

[Service]
WorkingDirectory=/path/to/MyHWManager/server/backend
ExecStart=/path/to/MyHWManager/server/backend/myhwmanager_backend
Restart=always
EnvironmentFile=/path/to/MyHWManager/server/backend/.env

[Install]
WantedBy=multi-user.target
```

```bash
sudo systemctl enable myhwmanager
sudo systemctl start myhwmanager
```

## Known Issues & Limitations

1. **CORS origins are hardcoded.** If you deploy on a non-localhost address, you must edit `main.go`. This should be moved to `.env`.
2. **Remote Desktop latency.** The current implementation relays base64-encoded frames over WebSocket. It works for admin tasks but is far from real-time. A proper WebRTC peer-to-peer implementation would be the fix.
3. **Terminal echo bug.** Occasionally, characters may double-echo in the remote terminal. This is a known PTY configuration issue that surfaces in certain shell configurations.
4. **Linux-only GPU detection.** The client uses `lspci` to detect GPUs, which doesn't exist on Windows/macOS. This needs platform-specific backends.
5. **No HTTPS by default.** All traffic is unencrypted. Use a reverse proxy for any non-localhost deployment.
6. **Single admin account.** There's no multi-user admin system. One username/password pair, configured in `.env`.
7. **SQLite limitations.** Fine for small deployments (< 50 devices). For larger scale, the database layer would need to be swapped to PostgreSQL.
8. **File Server storage.** Files are stored as flat files in `server/backend/storage/` with database-tracked metadata. There's no deduplication, compression, or quota system.
